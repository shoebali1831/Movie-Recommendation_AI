01 What I built

A full-stack AI-powered Movie Recommendation Engine. The user types a mood or feeling in plain English — like "something dark and psychologically intense" — and the app uses AI semantic search to return the three most relevant movies from a database of 30 films.

This is not keyword matching. The app converts both the movie descriptions and the user's query into 1536-dimensional vectors using OpenAI's embedding model. PostgreSQL with the pgvector extension then finds the three movies whose vectors are closest in meaning using cosine similarity.

06 The two API endpoints I built
POST
/api/seed
Loops through all 30 movies, calls OpenAI to embed each description, stores the 1536-float vector in PostgreSQL. Idempotent — deletes existing rows first. Returns: { seededCount: 30 }
POST
/api/recommend
Accepts { mood: "..." } — embeds the mood text, runs cosine similarity with pgvector's <=> operator against all 30 movies, returns top 3 as [{ title, genre, description, score }]. Score is 0–1.

The pgvector query at the heart of the app:

SELECT title, genre, description,
       1 - (embedding <=> @queryVec::vector) AS score
FROM   movies
ORDER  BY embedding <=> @queryVec::vector
LIMIT  3;

<=> is pgvector's cosine distance operator. 1 - distance = similarity score. The database orders by distance ascending (closest first) and returns the top 3.

09 How the AI part actually works

OpenAI's text-embedding-3-small model converts any piece of text into a list of 1536 floating-point numbers. These numbers are not random — they are mathematically structured so that text with similar meaning produces numbers that point in a similar direction in a 1536-dimensional space.

For example, "Interstellar" (a film about space, time, and family) and a query like "epic sci-fi with emotional depth" will produce vectors that are closely aligned. "Hereditary" (a horror film) will produce a very different vector and score much lower.

The angle between two vectors is measured by cosine similarity. If two vectors point in the same direction, the angle is 0° and similarity is 1.0 (100%). If they point in completely different directions, similarity is 0. pgvector computes this inside PostgreSQL using the <=> operator.

⚠
Important: All 30 movies must be embedded with the same model (text-embedding-3-small). Mixing vectors from different models breaks similarity results because the number spaces are incompatible — a vector from one model has no meaningful relationship to a vector from another.
10 Code review — questions & answers

These are the 5 questions from the assignment's code review, answered in plain English.

Q1

What is stored in the embedding column in your database? What type is it and what do the numbers mean?

The embedding column stores a list of 1536 decimal numbers — like [0.023, -0.417, 0.891, …] going on for 1536 values. The data type is vector(1536), which is a special type added by the pgvector extension.

These numbers are not random. When I send a movie description like "A masked vigilante fights a chaotic criminal mastermind…" to the OpenAI API, it reads and understands the meaning of the text and then converts that meaning into these 1536 numbers. Think of it like GPS coordinates — except instead of locating a place on a 2D map, these numbers locate the meaning of the text in a 1536-dimensional space.

Two movie descriptions that are similar in theme and tone will produce numbers that are close together in that space. Two completely different movies will produce numbers that are far apart. That closeness is exactly what we measure when recommending films.

Q2

Why is /api/recommend a POST request and not a GET request?

GET requests put their data in the URL — like /api/recommend?mood=something+funny. That works fine for short, simple values, but the user's mood input could be a long sentence or paragraph, and URLs have a length limit. Some browsers or servers will reject or cut off long URLs.

POST requests send their data in the request body, which has no practical length limit and is cleaner for structured data like JSON — { "mood": "…" }.

There is also a semantic reason: GET should be used for fetching existing data without changing anything. Our endpoint is doing something more like a computation — it is sending text to OpenAI and running a database query — which makes POST more appropriate even if nothing is being permanently saved.

Q3

What does the <=> operator do in your SQL query? What would happen if you removed the 1 - from the score calculation?

The <=> operator is pgvector's way of measuring the cosine distance between two vectors — basically, how different two pieces of text are in meaning. It gives a number between 0 and 1 where 0 means identical and 1 means completely different.

I need similarity, not distance — so I use 1 - distance to flip it. Now 1 means perfect match and 0 means nothing in common. That is what I show as the percentage score on the result cards.

If I removed the 1 -, the score would show the distance instead of the similarity. The worst matching movies would score 0.9+ and the best match would score near 0. The ORDER BY clause would still work (because sorting by distance ascending is the same as sorting by similarity descending), but the number displayed on each card would be backwards and confusing.

Q4

If the OpenAI API goes down while a user is searching, what happens in your app right now? What should ideally happen?

Right now: The .NET backend throws an exception when the OpenAI call fails, the request returns a 500 error, and the React frontend catches that error and shows the message: "Server error: 500". The user sees the error box in the UI — the app does not crash or show a blank screen.

Ideally: The backend should catch the OpenAI exception specifically and return a clear, friendly error — like "AI service is temporarily unavailable, please try again in a moment" — rather than a generic 500. The frontend could also add a retry button so the user does not have to retype their mood. For production, you would add exponential backoff (try again automatically with increasing delays) and potentially a fallback to a simpler keyword search if the AI is down.

Q5

If you switched from text-embedding-3-small to text-embedding-3-large halfway through, why would your search results break — even if the new model is "better"?

Imagine two different people invented two different measuring systems for "meaning" — one uses metres, the other uses miles. Even if one ruler is more precise, you cannot compare a measurement from one system to a measurement from the other system. They are not compatible.

That is exactly what happens with different embedding models. The 30 movies in my database were embedded using text-embedding-3-small, which maps text into a particular mathematical space. If I switch to text-embedding-3-large for the user's query, it maps the query into a completely different mathematical space. Comparing the two is meaningless — the cosine similarity numbers would be random noise.

The fix is to re-embed all 30 movies with the new model first — delete all existing rows, run /api/seed again — so everything lives in the same space. Only then will comparisons be valid. Also note: text-embedding-3-large produces 3072-dimensional vectors, so the database column would need to change from vector(1536) to vector(3072) as well.

BACKEND

Program.cs

The brain of the entire backend — dependency injection, CORS, and both API endpoints live here.

~150 lines

What it does: Instead of multiple controller files, this uses .NET's Minimal API pattern — everything is in one file. It registers EmbeddingService as a singleton (one shared instance), sets up CORS to allow the React app on port 5173 to call the API, then defines two endpoints.

// --- Dependency Injection ---
var connectionString = builder.Configuration.GetConnectionString("DefaultConnection");
var apiKey           = builder.Configuration["OpenAI:ApiKey"];
builder.Services.AddSingleton(new EmbeddingService(apiKey));

// --- CORS for React on port 5173 ---
builder.Services.AddCors(options => options.AddPolicy("AllowFrontend",
    p => p.WithOrigins("http://localhost:5173").AllowAnyMethod().AllowAnyHeader()));

// --- POST /api/seed ---
app.MapPost("/api/seed", async (EmbeddingService embedder) => {
    using var conn = new NpgsqlConnection(connectionString);
    await conn.ExecuteAsync("DELETE FROM movies");      // clear old rows
    foreach (var movie in MovieData.Movies) {
        var vec = await embedder.GetEmbeddingAsync(movie.Description); // call OpenAI
        await conn.ExecuteAsync("INSERT INTO movies (title,genre,description,embedding) VALUES (@T,@G,@D,@V::vector)",
            new { T=movie.Title, G=movie.Genre, D=movie.Description, V=FormatVector(vec) });
    }
    return Results.Ok(new { seededCount = MovieData.Movies.Count });
});

// --- POST /api/recommend ---
app.MapPost("/api/recommend", async (RecommendRequest req, EmbeddingService embedder) => {
    var queryVec = await embedder.GetEmbeddingAsync(req.Mood);  // embed user mood
    using var conn = new NpgsqlConnection(connectionString);
    var results = await conn.QueryAsync<MovieResult>("""
        SELECT title, genre, description,
               1 - (embedding <=> @queryVec::vector) AS score
        FROM   movies
        ORDER  BY embedding <=> @queryVec::vector
        LIMIT  3
    """, new { queryVec = FormatVector(queryVec) });
    return Results.Ok(results);
});

// --- Helper: float[] → "[0.1,0.2,...]" for PostgreSQL ---
static string FormatVector(float[] v) =>
    "[" + string.Join(",", v.Select(x => x.ToString("G9", CultureInfo.InvariantCulture))) + "]";

// --- Request DTO ---
record RecommendRequest(string Mood);
ℹ
Why FormatVector? Npgsql can handle the vector type natively for reading, but for parametrised inserts we need to pass it as the PostgreSQL literal string format [0.1,-0.3,...] and cast it with ::vector. FormatVector converts a C# float[] into that exact string.
BACKEND / AI

models/Services/EmbeddingService.cs

Wraps the OpenAI SDK — the only file that talks to the AI.

~20 lines

What it does: Creates an EmbeddingClient using the text-embedding-3-small model. When GetEmbeddingAsync(text) is called, it sends the text to OpenAI and gets back 1536 floats. That array is what gets stored in (or compared against) PostgreSQL.

public class EmbeddingService
{
    private readonly EmbeddingClient _client;

    public EmbeddingService(string apiKey)
    {
        _client = new OpenAIClient(apiKey)
            .GetEmbeddingClient("text-embedding-3-small");
    }

    public async Task<float[]> GetEmbeddingAsync(string text)
    {
        ClientResult<Embedding> result = await _client.GenerateEmbeddingAsync(text);
        return result.Value.ToFloats().ToArray();  // 1536 floats
    }
}
ℹ
Why a separate class? So that Program.cs does not import OpenAI SDK directly. It also makes it easy to swap the model or provider later — only this one file needs to change.

What it does: Manages 4 pieces of state: mood (what the user typed), results (array of 3 movies), loading (boolean for spinner), and error (error message string). When the user clicks the button, handleSearch() runs. It shows suggestion chips to help users who don't know what to type.

const [mood,    setMood]    = useState('');
const [results, setResults] = useState([]);
const [loading, setLoading] = useState(false);

async function handleSearch() {
  setLoading(true); setError(null); setResults([]);
  try {
    const res = await fetch("http://localhost:5088/api/recommend", {
      method: "POST",
      headers: { "Content-Type": "application/json" },
      body: JSON.stringify({ mood }),
    });
    if (!res.ok) throw new Error(`Server error: ${res.status}`);
    setResults(await res.json());    // [{title, genre, description, score}, ...]
  } catch (e) {
    setError(e.message);
  } finally {
    setLoading(false);
  }
}
FRONTEND

src/MovieCard.jsx

The result card component — one card per returned movie.

~40 lines

What it does: Receives movie (title, genre, description, score) and rank (0, 1, 2) as props. Uses rank to pick the medal emoji and gold/silver/bronze colour. Renders an animated score bar — the bar width is set via a CSS custom property --width which is driven by the score percentage.

const MEDALS      = ['🥇', '🥈', '🥉'];
const RANK_COLORS = ['#f5a623', '#c0c0c0', '#cd7f32'];  // gold / silver / bronze
const RANK_LABELS = ['BEST MATCH', '2ND MATCH', '3RD MATCH'];

export function MovieCard({ movie, rank }) {
  const pct = Math.round(movie.score * 100);        // 0.41 → 41%
  const accent = RANK_COLORS[rank];
  return (
    <div className="movie-card" style={{ '--accent': accent }}>
      <div className="rank-row">
        <span className="medal">{MEDALS[rank]}</span>
        <span className="rank-label">{RANK_LABELS[rank]}</span>
      </div>
      <h2>{movie.title}</h2>
      <p className="genre">{movie.genre}</p>
      <p className="desc">{movie.description}</p>
      <div className="score-bar-wrap">
        <div className="score-bar" style={{ '--width': `${pct}%` }} />
        <span className="score-label">{pct}% match</span>
      </div>
    </div>
  );
}
FRONTEND / STYLES

src/App.css

The full dark glassmorphism visual theme.

What it does: Defines the entire visual identity of the app. Key techniques used:

ANIMATED ORBS

Three fixed-position blurred circles (purple, blue, pink) with a drift keyframe animation that slowly moves them around the background.

GLASSMORPHISM

Cards use background: rgba(255,255,255,0.05) + backdrop-filter: blur(12px) to create the frosted glass look over the animated background.

ANIMATED SCORE BAR

The score bar starts at width: 0 and animates to var(--width) using a CSS transition on mount — the width value is injected from React as an inline style.

COMPACT HEADER

After the first search, the .searched class is added to the container, which triggers a CSS transition shrinking the header so result cards have more room.
